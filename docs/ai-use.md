# How AI was used on this project

> **The honest headline:** nearly all of the code in this repository was generated by an AI
> agent (Claude Code). We directed it, reviewed it, corrected it, and can explain it. This
> document says which mode we were operating in for which part, and where that broke down.
>
> Companion documents: [`PROVENANCE.md`](../PROVENANCE.md) records origin and review depth
> per module, plus our own explanation of each correctness-critical one;
> [`architecture.md`](architecture.md) explains what the system is.

---

## 1. The modes

| Mode | What it means here |
|---|---|
| **Operator / Agent** | The AI does the work. We specify intent and constraints, the agent writes the code, and we accept, revise, or reject the result. Our contribution is the specification and the judgment, not the keystrokes. |
| **Critic** | The AI's output is the thing under review. We read it, question it, test it, find what's wrong, and send it back. Our contribution is the correction. |

Most stages used both, in that order: the agent produced a stage, then we reviewed it before
the next stage started. That ordering is visible in the git history (§3).

### Written wholesale by the agent, accepted with light review

`models.py`, `data_loader.py`, `checklist.py`, `main.py`, the ingestion package, the frontend
components, the devcontainer, the Makefile, and CI. These are conventional code where a
mistake is visible and cheap: if the checklist renders wrong, you see it.

### Heavily revised by us after the agent produced them

- **The prereq checklist** — the agent's first version showed the wrong scope of courses. We
  reworked what it selects across `90ebd4b`, `52db4dd`, and `47800c8`.
- **The frontend calendar/UI** — revised for intuitiveness in `b8af3c5`; we moved prereq
  confirmation out of the chat and into a panel beside the calendar because the agent's
  original placement buried it.
- **The LLM's role** — the agent's initial design had the LLM "explain and rank" schedules. We
  narrowed it twice: first to a provider-agnostic interface, then to *propose constraints
  only*. The narrowing was our call each time.

### Reviewed line-by-line, because a defect is a wrong schedule

`prereq.py`, `solver.py`, `ranking.py`, `verifier.py`, `orchestrator.py`, `requirements.py`,
`requirements_loader.py`, `rationale.py`. Our explanations of these are in
[`PROVENANCE.md`](../PROVENANCE.md) — that section exists specifically as evidence we can
explain the code, not merely that it exists.

## 4. Where being a Critic actually caught something

Three examples, because "we reviewed it" means nothing without them.

### 4.1 Completed courses were being recommended back

The agent's solver satisfied prerequisites from the student's completed set — and then happily
scheduled those same completed courses. Both roles of `completed_courses` were conflated. A
human noticed it against real output (`abcd5f8` "checked the error that was happening with
ticking prereq boxes"), and it was fixed in `b8ac1e8` with a regression test that still runs.

### 4.2 Duplicate schedules — the one that matters most

The recommender was returning five "options" of which four were **byte-identical**. The cause
was in the solver's DFS: it offered the same section set at every skip node, so the top-K heap
filled with copies.

This is the most instructive failure in the project, because of what *didn't* catch it:

- The agent wrote the solver.
- The agent also wrote the solver's tests — property-style, over hundreds of random
  scenarios, checking no conflicts, commitments respected, cap honored, at most K results.
- **Every one of those tests passed.** The bug shipped anyway.

It was caught by a human looking at the UI and saying *these all look the same*. The lesson we
took: **tests written by the same agent that wrote the code inherit its blind spots.** The
agent tested every property it thought of; "the options should differ from each other" was not
one, because it never occurred to the agent that they might not. No amount of agent-generated
coverage substitutes for a person checking the output against what the product is *for*.

The fix added `schedule_key()` and de-duplication at the source, plus the test that should have
existed from the start: no two returned schedules are identical.

### 4.3 The Bedrock assumption

The original design specified Amazon Bedrock for LLM serving. That was a reasonable choice on
paper and would have been undeployable for us: AWS Academy Learner Lab may not have Bedrock
enabled at all. The constraint came from us, not the agent — the agent had no way to know the
deployment environment's limits. It implemented the provider abstraction we asked for.
